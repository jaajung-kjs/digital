# 조직 트리 → 워킹카피 통합 (B) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development. Steps use checkbox (`- [ ]`) syntax.

**Goal:** 본부/지사/변전소/층 CRUD를 즉시-API에서 전역 워킹카피 staged + 단일 `/commit`으로 전환(레벨 간 temp-id 재매핑, 새 변전소+층+자산 원샷 커밋).

**Architecture:** 조직 4계층을 WC 컬렉션으로 추가. 앱 시작 시 전체 조직트리(메타) eager 로드 → 트리는 effective 투영. CRUD는 `wc.stage*`(삭제는 하위까지 cascade stage-delete). 커밋은 기존 전역 `/commit` 확장(OCC·idMaps·위상정렬 재사용).

**Tech Stack:** Express+Prisma(zod), React+Zustand+RQ, Vitest. 백엔드 빌드 `cd backend && npm run build`, 프론트 `cd frontend && npm run build`, 테스트 각 `npx vitest run`(백엔드 테스트 러너 확인).

**근거 스펙:** docs/superpowers/specs/2026-06-17-조직트리-워킹카피-통합-design.md

**원칙:** 기존 패턴을 *그대로* 확장(assets/cables 처리 미러). 새 코드 최소, scatter 삭제. CONVENTIONS 준수(프론트 UI).

## Phase 3 — 백엔드: 커밋 서비스 확장

### Task 3: 조직 엔티티 OCC + 위상정렬 create/update/delete + 자산 substationId/floorId 해소

**Files:** Modify `backend/src/services/substationCommit.service.ts`

먼저 `run()` 전체(OCC Phase, idMaps, asset create/update, orderCreatesByParent, loadOcc, collectConflicts, CommitResult)를 읽고 동일 패턴으로 확장.

- [ ] **Step 1 (테스트 우선):** 커밋 통합 테스트 추가(백엔드 테스트 관례 확인 후). 핵심 케이스:
```ts
// 새 HQ→branch→substation→floor→asset 한 페이로드, 모두 tempId 상호참조
// commitSubstation('', payload, userId) 호출 후:
//  - idMaps.headquarters/branches/substations/floors/assets 채워짐
//  - 생성된 asset.substationId === 생성된 substation real id, asset.floorId === 생성된 floor real id
//  - DB에 4계층 + asset 존재
// (실제 DB 또는 prisma 모킹 — 기존 substationCommit 테스트 방식 따름. 없으면 최소 통합 테스트 신설.)
```
실제 prisma 통합 테스트가 무거우면, 위상정렬·idMaps 해소를 담당하는 순수 헬퍼를 분리해 단위 테스트(아래 Step 3에서 `orderByParent` 제너릭).
- [ ] **Step 2: OCC 확장** — assets/cables OCC 블록 바로 아래에 4종 추가(각각 `loadOcc`+`collectConflicts` 재사용; Substation/Branch/Floor/HQ는 `updatedAt` 보유). headquarters/branches/substations/floors의 updates+deletes id 로드 → baseVersion 비교 → conflicts.
- [ ] **Step 3: 생성 — 위상정렬 + idMaps** — asset create *이전*에 4단계 순차:
```ts
idMaps.headquarters = {}; idMaps.branches = {}; idMaps.substations = {}; idMaps.floors = {};
// HQ
for (const c of (input.headquarters?.creates ?? [])) {
  const r = await tx.headquarters.create({ data:{ name:c.name, sortOrder:c.sortOrder??0, createdById:userId, updatedById:userId }});
  idMaps.headquarters[c.tempId] = r.id;
}
// branch (hqId 해소)
for (const c of (input.branches?.creates ?? [])) {
  const hqId = idMaps.headquarters[c.headquartersId] ?? c.headquartersId;
  const r = await tx.branch.create({ data:{ headquartersId:hqId, name:c.name, sortOrder:c.sortOrder??0, createdById:userId, updatedById:userId }});
  idMaps.branches[c.tempId] = r.id;
}
// substation (branchId 해소)
for (const c of (input.substations?.creates ?? [])) {
  const branchId = c.branchId ? (idMaps.branches[c.branchId] ?? c.branchId) : null;
  const r = await tx.substation.create({ data:{ branchId, name:c.name, address:c.address??null, sortOrder:c.sortOrder??0, createdById:userId, updatedById:userId }});
  idMaps.substations[c.tempId] = r.id;
}
// floor (substationId 해소)
for (const c of (input.floors?.creates ?? [])) {
  const subId = idMaps.substations[c.substationId] ?? c.substationId;
  const r = await tx.floor.create({ data:{ substationId:subId, name:c.name, floorNumber:c.floorNumber??null, sortOrder:c.sortOrder??0, createdById:userId, updatedById:userId }});
  idMaps.floors[c.tempId] = r.id;
}
```
(각 레벨은 단일 부모라 입력 순서대로 처리하면 충분 — 같은 레벨 내 부모-자식 없음. 부모 tempId는 이전 레벨 idMaps로 해소되므로 별도 위상정렬 불필요.)
- [ ] **Step 4: update/delete** — 4종 각각: updates는 부분 패치(부모 FK 패치 시 idMaps 해소) + `updatedById`; deletes는 `deleteMany({where:{id:{in:[...]}}})`(DB cascade가 하위 처리). 응답 `updated.{hq,branches,substations,floors}` 수집(touched id의 updatedAt).
- [ ] **Step 5: 자산 substationId/floorId tempId 해소** — asset create: `substationId = c.substationId ? (idMaps.substations[c.substationId] ?? c.substationId) : substationId`; `floorId = c.floorId ? (idMaps.floors[c.floorId] ?? c.floorId) : null`. asset update: patch에 substationId/floorId 있으면 동일 해소.
- [ ] **Step 6: CommitResult** 인터페이스에 idMaps/updated 4종 추가.
- [ ] **Step 7:** `cd backend && npm run build` → 0. 테스트 실행(있으면) 그린.
- [ ] **Step 8:** Commit `feat(커밋): 조직 엔티티 staged 커밋 — OCC·tempId 재매핑(HQ→지사→변전소→층→자산)`.
