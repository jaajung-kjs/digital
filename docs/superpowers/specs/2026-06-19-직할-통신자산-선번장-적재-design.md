# 강원본부 직할 통신자산 · 선번장 실데이터 적재 설계

**작성일:** 2026-06-19
**상태:** 설계 (브레인스토밍 합의 완료)
**범위:** 강원본부 직할 13개 국소의 통신설비/광케이블 실데이터를 엑셀 6종에서 추출해 현재 자산 DB(Asset/Cable 범용 모델)에 적재. **레이어 ①물리 광백본 + ②설비** 까지. 배치(좌표)·레이어 ③논리회선은 범위 밖.

---

## 1. 목표

기존 가짜 시드(`gangwonSubstations.ts`의 OFD/랙/모듈 더미)를 제거하고, `직할/` 폴더 엑셀 6종 기반의 **실제 변전소·통신설비·광케이블 연결**로 교체한다. 결과: 13개 국소의 OFD·통신장비가 자산으로, 변전소간 OPGW와 OFD 코어(선번)가 케이블로 들어가 선번장 그리드·연결탭·토폴로지에서 실데이터로 보인다.

## 2. 확정된 결정 (브레인스토밍)

1. **범위 = ①물리 광백본 + ②설비.** 레이어 ③(설비-설비 논리회선: E1/T1·보호·SCADA 멀티홉)은 **다음 슬라이스**.
2. **신규 AssetType은 필요한 만큼 생성**(§6). 기존에 있으면 재사용.
3. **기존 가짜 시드 전부 삭제 후 실데이터로 교체**. 첫-배포 가드(빈 DB에만 시드, [[project_deploy_seed_idempotency]]) 유지.
4. **필드는 전부 JSON(데이터 주도) 저장, 스키마 컬럼 추가 0건.** `Asset.attributes`, `Cable.specParams`. 마이그레이션 없음.
5. **편집/표시는 "기존 편집기 범위"만(결정 A).** §4.1 참조 — 신규 메타필드는 v1에서 *시드 저장 전용*, 인앱 편집/표시는 후속 슬라이스.
6. **외부 상대국(홍천·청평·가평·강원DD 등 13직할 밖)은 스텁 자산**으로 생성해 케이블 양끝 보존.

## 3. 현재 대상 모델 (확인됨 — 대통일 P3a, Cable 기반)

별도 FiberPath/FiberCore 테이블 없음(6-13 설계문서는 stale). 선번장 = "허브+번호슬롯" 범용 패턴:

- **OFD 자산**: `assetType.placementKind==='OFD'`, substationId 보유.
- **OFD-SLOT 자산**: `code==='OFD-SLOT'`, `connectionKind==='conduit'`, `parentAssetId=OFD`, `name=대국명`. 양 변전소에 각각.
- **OUT 케이블(=코어 1개)**: `cableType='FIBER'`, `sourceRole/targetRole` 한쪽 `'OUT'`, `number=코어번호`, 한끝=slot, 다른끝=로컬 설비. `specParams={purpose, circuitText, spliceType, usageOverride, ...측정메타}`.
- **OPGW 케이블(변전소간 백본)**: `cableType='FIBER'`, `sourceRole='IN' & targetRole='IN'`, 양끝=두 slot, `specParams.cores`=용량. substationId polymorphic(양 변전소 effectiveCables 노출).
- **JSON 메커니즘**(이미 사용 중): `AssetType.fieldTemplate`(예: 광설비 `OPT_FIELDS` = 대국·구성형태·링명칭·규격·도입년도·IP주·IP확장) + `Asset.attributes`. `CableCategory.specTemplate` + `Cable.specParams`. **컬럼 추가 불필요.**

## 4.1 편집/표시 범위 (결정 A — 우선 적재, 편집기 후속)

현 앱은 템플릿으로 폼을 **자동 생성하지 않는다**(확인됨). 편집/표시 가능 영역은 고정:
- **자산 편집 가능**: 스칼라 컬럼 8개만 — `name·installDate·manager·description·status·warrantyUntil·replaceDue·sortOrder`. (`ASSET_SCALAR_FIELDS`, 커밋 경로 SSOT.)
- **자산 종류별 속성**(`fieldTemplate`→`attributes`): 렌더러는 있으나 **#7에서 비활성**(attributes read/write 경로 제거) → 현재 **표시·편집 둘 다 안 됨**.
- **선번장 코어 편집 가능**: 그리드 하드코딩 4컬럼 — `용도(purpose)·수용내역(circuitText)·융착(spliceType)·사용(usageOverride)` (`fiberRegisterDescriptor`).
- **그 외 specParams 키**(손실·코어수·단면적 등): 저장되나 그리드에 컬럼 없으면 표시·편집 안 됨.

**v1 적재 정책:**
- **인앱에서 보이고 편집되는** 데이터로 핵심을 싣는다: 변전소·층·자산(이름/종류/manager/installDate/status), 선번장 코어(용도·수용내역·융착·사용), OPGW·코어 케이블 연결(연결탭·토폴로지·트레이스).
- **부가 메타**(설비 model·maker·IP·λ·generation, 코어 loss1310/1550·점검, OPGW cores·긍장·단면적)는 `attributes`/`specParams`에 **저장만**(시드 전용). 표시·편집은 후속(자산 속성폼 재활성 / 선번장 컬럼 추가 = 별도 슬라이스).
- 따라서 v1 산출물의 가시 가치 = "13 국소의 통신자산 + 선번장 + OPGW 토폴로지가 앱에서 보이고 추적된다". 풍부한 스펙 메타는 DB에 적재돼 후속 UI에서 노출 대기.

## 5. 적재 아키텍처 (3단계 — 추출/검수/시드 분리)

엑셀은 컨테이너에 없고 파싱이 지저분(병합셀·도형·이미지·오타) → **오프라인 추출 → 검수된 JSON → 결정적 시드** 로 분리.

1. **오프라인 추출(개발기, 1회)**: python(openpyxl; .xls는 olefile/TXO) 스크립트가 6종 엑셀 → **정규화 JSON** 산출. 위치: `backend/prisma/seed/data/jikhal/{substations,assets,ofdCores,opgw,externalStubs}.json`. 스크립트는 `scripts/extract-jikhal/`.
2. **사람 검수**: 모호/판독 항목 교정 — (구)/(신)춘천 구분, RTU 링(이미지) 노드·IP, 명칭 alias, 양방향 dedup, "(예)/현장확인" 제외. JSON은 diff·리뷰 가능.
3. **시드 모듈** `backend/prisma/seed/jikhalAssets.ts`: 검수 JSON을 읽어 결정적 UUID로 Substation/Floor/Asset/Cable 생성. `gangwonSubstations.ts` 대체. 첫-배포 가드 안에서 호출.

**결정적 id**: 기존 `gwUuid(type,idx)` 패턴 유지(`9a0000TT-0000-4000-b000-IIIIIIIIIIII`, floor plan API의 uuid 검증 통과). type 슬롯: sub/floor/ofd/slot/asset(설비)/opgwCable/outCable/externalStub.

## 6. 신규 AssetType + fieldTemplate

**기존 재사용**: OFD, OFD-SLOT, RACK, OPT-COT(통합단말), OPT-SMALL(소형광), OPT-TRANS·EQP-OPT-TERM(송변전광단말), PCM, PITR·EQP-PITR-2000·EQP-PITR-5000, RTU·EQP-RTU·EQP-SCADA, EQP-NET-SW.

신규 종류 자체(트리/배치/연결의 종류 분류)는 v1에 필요. 단 `fieldTemplate` 정의는 **후속 편집기용 선언**이며(§4.1 — attributes 폼 비활성), v1에선 해당 값을 `attributes`에 시드 저장만 한다.

**신규 코드(assetTypes.ts 추가, group='통신', fieldTemplate JSON):**
| code | name | fieldTemplate(주요 키) |
|---|---|---|
| `PIU` | 보호용단말(SaTOP) | model, maker, tlName, counterpart |
| `SPS` | 특수보호장치 | model, maker, counterpart |
| `DAS` | 배전자동화 | ringName, circuit |
| `KEPCIT` | 권역망전송장치 | generation(차세대/구), ringName, circuit |
| `POWERDUCT` | 전력구감시 | circuit |
| `OPT-SWITCH` | 광스위치(예비링) | ipAddress, ringNode, maker, spec |
| `OPT-CONV` | 광컨버터 | spec |
| `MUX` | 다중화장비 | model(PTPU/OMS/CMUX/MSPP/I-MUX), maker | (또는 기존 PCM 흡수 — 추출 시 확정) |

보드(CTRU/PIU/LIU)·통합단말 내부 DS슬롯·다중화 채널은 **레이어 ③ 범위** → v1은 설비 본체만, 보드/채널은 `attributes`에 요약 보관(자산 분해는 후속).

PITR 모델 변형(2000/5000/S/D, PIU, SPS)은 `EQP-PITR-2000/5000` 재사용 + `attributes.model/generation/board`. (별도 코드 남발 금지.)

## 7. 신규 광 CableCategory + specTemplate

`cableCategories.ts`에 광(displayGroup='광') 카테고리 추가(없으면). specTemplate:
- **OPGW(백본)**: `cores`(총코어), `lengthKm`(긍장/실장), `sectionArea`(SQ), `cableKind`(OPGW/OFC/FOFC), `installMethod`(가공/지중), `strands`(조수), `startTower/endTower`(호번).
- **OUT(코어)**: `purpose`(유형), `circuitText`(수용내역), `spliceType`(융착/패치/단선), `usageOverride`(사용/미사용), `loss1310/loss1550`(dB), `dist1310/dist1550`(km), `inspectResult`(적합/부적합/측정불가), `inspectDate`.

전부 specParams(JSON). 컬럼 추가 없음. **편집/표시**: OUT 코어의 `purpose·circuitText·spliceType·usageOverride`만 선번장 그리드에서 편집 가능(기존 4컬럼). 나머지(loss·cores·긍장·단면적 등)는 시드 저장 전용 — 선번장 컬럼 추가는 후속(§4.1).

## 8. 매핑 규칙

**자산:**
- 13 국소 → Substation(실제명). Floor='통신실' 1개.
- 국소별 OFD 1개(코어가 다수 OFD로 갈리면 추출 시 분리). 대국별 OFD-SLOT(name=정규화 대국명).
- 설비(파일 ②): 국소별 통합단말·소형광·송변전광단말·PITR(들)·RTU·다중화·광스위치 등 — 파일에서 **dedup된 물리장비 1대=Asset 1개**. 양방향/양시트 중복 제거.

**케이블:**
- **OPGW**: 직할쌍마다 두 OFD-SLOT을 IN/IN으로 연결. specParams=코어수·긍장·단면적 등. dedup(양방향 1개).
- **OUT(코어)**: OFD선번장 각 코어 → 해당 국소 slot ↔ **유형에 해당하는 로컬 설비**로 OUT 케이블. number=코어번호. specParams=용도/수용내역/융착/사용/손실.
  - 한 유형에 설비가 여럿이면 `회선수용내역` 텍스트로 매칭 시도, 실패 시 그 유형 대표 설비에 연결 + `attributes.needsReview=true`.
  - 유형이 설비 아님(임대·기타)이면 설비 없이 slot 점유 메타만(OUT 케이블의 far end는 스텁 또는 생략 — 추출 시 정책).

**정규화(필수, alias 맵):** `춘천S/S`↔`(구)/(신)춘천`, `H/P`↔`HP`, `S/Y`↔`열병합`, `춘천열병합`↔`열병합`, 공백/`#n`/`(좌측N열)`/`비금속`/`경로A/B` 제거. **(구)/(신)춘천 모호**는 검수에서 확정.

## 9. 외부 노드 / 중복 / 미확정 처리

- **외부 상대국**(13직할 밖): `externalStub` 타입 스텁 Substation 또는 스텁 Asset 1개씩 생성(이름만), OPGW/코어 케이블 양끝 보존. 토폴로지 완전. (스텁엔 통신실/설비 안 만듦.)
- **양방향 중복**: OPGW·회선이 양 국소 시트에 각각 → (국소쌍+T/L+코어) 키로 dedup, 1개만 생성.
- **미확정 행**: `(예)`/`예비`/`현장확인 필요`/빈 포트 → v1 미적재 또는 `usageOverride='미사용'`. 단선(융착) 코어는 `spliceType='단선'`+미사용.
- **RTU 예비링**: 이미지 판독 결과(노드·IP·OFD포트)는 검수 JSON에 수기 입력.

## 10. 시드 교체 (gangwonSubstations → jikhalAssets)

- `seed.ts`의 `seedGangwonSubstations` 호출을 `seedJikhalAssets`로 교체. 첫-배포 가드(`headquarters.count()===0`) 안에 유지.
- 새 코드/AssetType·CableCategory(참조데이터)는 가드 밖(매 배포 upsert) — assetTypes/cableCategories 시더에 추가.
- 기존 `gangwonSubstations.ts`는 삭제(가짜). 결정적 id라 첫 배포 외 재실행 없음.

## 11. 범위 밖 (후속 슬라이스)

- **레이어 ③ 논리회선**: 설비-설비 E1/T1·보호(SAToP)·SCADA·배전전력구 멀티홉 회선(송변전광단말/소형광/PITR/RTU 회선경로). 보드/채널 자산 분해.
- **배치 좌표**(positionX/Y): 나중 수동/자동 배치.
- `강원DD`·`전송손실`·D/O 판매사업소 링·765/345kV 광역 T/L·강릉/동해/태백/원주권 전체.

## 12. 테스트 / 검증

- **추출 스크립트**: 산출 JSON 행수·alias 미해소·dedup 후 카운트를 로그. 단위테스트는 정규화 함수(alias·dedup) 위주.
- **시드**: 빈 dev DB에 시드 후 psql로 검증 — 13 국소, OFD/slot/설비 자산 수, OPGW/OUT 케이블 수, FIBER 케이블 양끝 무결성(slot↔slot, slot↔설비), 고아 케이블 0.
- **앱**: 한 국소 선번장 그리드에 코어가 유형·수용내역과 함께 뜨고, 연결 클릭 시 토폴로지 추적(cableTrace)이 대국 OFD까지 가는지.
- 백/프론트 빌드·기존 테스트 그린.

- **추출 정확도**: 병합셀/도형(.xls)/이미지(RTU링)/오타 → 검수 단계가 품질 게이트. JSON diff로 검증.
- **코어→설비 매칭 모호**: 동일 유형 다수 설비 시 휴리스틱 → `needsReview` 플래그로 표시, 후속 교정.
