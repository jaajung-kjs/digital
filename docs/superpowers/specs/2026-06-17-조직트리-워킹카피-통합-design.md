# 조직 트리 → 워킹카피(staged + 단일 커밋) 통합 설계 (B)

**작성일:** 2026-06-17
**상태:** 설계 승인됨(B, 1=a, 2=i) → 구현 계획
**범위:** 본부/지사/변전소/층 CRUD를 즉시-API에서 **전역 워킹카피 staged + 단일 `/commit`** 으로 전환. 새 변전소+층+자산을 한 트랜잭션에 커밋(레벨 간 temp-id 재매핑).

## 1. 목표

조직 트리 CRUD를 앱의 git-like 모델(staged → 리뷰/undo → 단일 커밋)에 일치시킨다. 즉시-API + 수동 store/RQ 동기화(scatter)를 제거하고, 조직 엔티티를 WC 컬렉션으로 통합한다.

## 2. 현재 상태 (확정 사실)

- **커밋은 이미 전역**: `POST /api/commit` → `substationController.commitGlobal` → `commitSubstation('', body, userId)`. 각 create가 자기 `substationId`를 싣고, `idMaps`(tempId→realId)·`orderCreatesByParent`(위상정렬)·`loadOcc`/`collectConflicts`(OCC, RepeatableRead 트랜잭션)로 처리. (커밋 스코프 재설계 불필요.)
- **WC 엔진은 제너릭**: `COLLECTIONS={assets,cables,records}` + `makeDescriptor`(idOf/versionOf) + `mergeEffective` + 제너릭 `put/patch/remove` + `sumOverlaysDirty`. 컬렉션 등록만으로 확장.
- **조직 트리는 분리**: `organizationApi`(listHeadquarters + 자식 lazy) → `organizationStore`. CRUD는 즉시-API(`useOrgNodeCrud`). 전체 트리 엔드포인트 없음.
- 자산 커밋은 `floorId`/`substationId`의 tempId를 **해소하지 않음**(기존 id만 가정).

## 3. 아키텍처

조직 엔티티(headquarters/branches/substations/floors)를 **WC 컬렉션**으로 추가한다. 트리는 WC effective의 평면 리스트에서 트리를 구성해 렌더(다른 화면과 동일한 effective 투영). CRUD는 `wc.stage*`로 staged → 단일 `/commit`으로 영속. **앱 시작 시 전체 조직트리(메타만)를 한 번 로드**(결정 1=a).

```
앱 시작 → GET /organizations/tree → WC.saved.{headquarters,branches,substations,floors}
트리 = mergeEffective(saved, overlay) 평면 리스트 → 부모 포인터로 트리 구성(펼침 상태는 로컬 UI)
CRUD → wc.stage{Create,Update,Delete}('headquarters'|'branches'|'substations'|'floors', …)  (즉시-API 제거)
삭제 → 클라이언트가 하위까지 stage-delete(트리 워크, 결정 2=i)
저장 → 기존 단일 커밋(/commit)에 org 델타 포함 → 백엔드가 HQ→지사→변전소→층→자산 순 temp-id 재매핑
```

## 4. 백엔드 변경

### 4.1 전체 트리 엔드포인트 (신규)
`GET /api/organizations/tree` → `{ headquarters: HqDTO[], branches: BranchDTO[], substations: SubDTO[], floors: FloorDTO[] }` (각 평면 배열, 부모 FK·updatedAt 포함). 조직 메타만(자산 아님). 조회는 비인증(기존 list와 동일). 새 컨트롤러/서비스 메서드(기존 list 재사용/조합 가능).

### 4.2 커밋 스키마 (`substationCommit.schema.ts`)
`headquarters/branches/substations/floors` 4개 `collection(create, patch)` 추가. create는 `tempId` + 필드(+ 부모 FK는 tempId 또는 real id 허용: branch.headquartersId, substation.branchId, floor.substationId). patch는 `.partial()`. `substationCommitSchema`에 4개 optional 추가. 기존 `floor`(단일, 캔버스 설정 OCC) 섹션은 **유지**(에디터 도면설정 커밋용 — org `floors` 컬렉션과 별개 관심사).

### 4.3 커밋 서비스 (`substationCommit.service.ts`)
- `CommitResult.idMaps/updated`에 4종 추가.
- **OCC**: 4 컬렉션 update/delete의 baseVersion 검증(`loadOcc`/`collectConflicts` 재사용).
- **생성 순서(위상정렬)**: HQ → branch(hqId 해소) → substation(branchId 해소) → floor(substationId 해소). 각 레벨은 단일 부모라 `orderCreatesByParent`를 제너릭화하거나 레벨 순차 처리. idMaps에 tempId→realId 기록.
- **자산 tempId 해소 확장**: asset create/update에서 `substationId`/`floorId`를 `idMaps.substations`/`idMaps.floors`로 해소(신규 변전소/층 참조 가능). → 새 변전소+층+자산 원샷 커밋 성립.
- update/delete는 부분 패치 + deleteMany(cascade는 DB onDelete:Cascade가 처리 — 부모 delete 시 하위 자동).
- 트랜잭션·OCC·응답 패턴은 기존 그대로.

## 5. 프론트엔드 변경

### 5.1 WC 컬렉션 추가 (`substationStore.ts` + `hooks.ts`)
- `SavedCollections`/`Overlays`에 `headquarters/branches/substations/floors` 추가. `COLLECTIONS` 레지스트리에 4종 descriptor 등록(제너릭 makeDescriptor). `buildSaved`가 4종 평면 배열을 담음.
- `effectiveHeadquarters()/effectiveBranches()/effectiveSubstations()/effectiveFloors()` 게터 + 훅(`useEffective*`). mergeEffective 재사용.
- `loadOrgTree()` 액션: `GET /organizations/tree` → saved 4종 세팅(+baseVersions). 앱 시작(AppShell/TreePanel mount)에서 1회.

### 5.2 트리 컴포넌트 (`TreePanel.tsx`)
- 데이터 소스를 `organizationStore`(lazy)에서 **WC effective 4종 평면 리스트 → 부모 포인터로 트리 구성**으로 전환. 펼침 상태는 로컬(useState/소형 UI store). 클릭=내비게이션·드래그 reorder·케밥 CRUD UI는 유지.
- `organizationStore`는 제거 또는 펼침-상태 전용으로 축소(데이터는 WC). (scatter 삭제 — 트리 데이터 이중화 해소.)

### 5.3 CRUD 재배선 (`useOrgNodeCrud.ts`)
- `addChild`/`addHeadquarters`/`rename`/`remove`를 즉시-API에서 **`wc.stageCreate/stageUpdate/stageDelete(컬렉션키, …)`** 로 교체. tempId는 `generateTempId()`. RQ 무효화/`fetchChildNodes` 재로드 로직 제거(effective가 자동 반영).
- **삭제 cascade(결정 2=i)**: 본부/지사/변전소 stage-delete 시 effective에서 하위(지사/변전소/층, 그리고 변전소면 그 변전소의 자산/케이블)를 트리 워크로 찾아 함께 stage-delete → 커밋 전 뷰 일관 + 커밋 델타 정확. (DB cascade와 중복돼도 안전.)

### 5.4 커밋 통합 (`substationCommit.ts` / `useCommitWorkingCopy.ts`)
- `buildSubstationCommitPayload`에 org 4종 델타 추가(`buildDelta` 재사용, create→tempId, update→baseVersion+patch, delete→baseVersion).
- 커밋 응답 idMaps로 saved 갱신(기존 패턴) + 커밋 후 `loadOrgTree()` 재로드(또는 응답 반영). dirty 집계·커밋바에 org 포함(`sumOverlaysDirty`가 자동 — 컬렉션 등록 시).

## 6. temp-id 재매핑 체인

create 순서 HQ→branch→substation→floor→asset. 각 단계 idMaps에 tempId→realId 기록, 다음 단계 부모 FK 해소. 자산의 substationId/floorId도 해소(4.3). 클라이언트 stage 시 부모로 tempId를 그대로 참조(예: 새 변전소(temp) 아래 새 층(temp, substationId=변전소 temp)) → 커밋이 일괄 해소.

## 7. 테스트

- **백엔드 단위/통합**: `/commit`에 org create(레벨 간 tempId 참조: 새 HQ→branch→substation→floor→asset) → 모두 생성·재매핑·FK 정합 확인. OCC 충돌(stale baseVersion) 409. delete cascade.
- **프론트 단위**: WC effective 4종 게터(mergeEffective), 트리 구성(평면→트리), `useOrgNodeCrud` stage 매핑(타입→컬렉션키, 하위 cascade stage-delete), buildSubstationCommitPayload org 델타.
- **빌드/기존 테스트** 그린. **브라우저 수동**: 본부/지사/변전소/층 추가·이름변경·삭제가 staged(저장 전 트리 반영, 커밋바 dirty↑) → 단일 저장 후 영속, 새 변전소+층+자산 원샷 저장, 삭제 cascade, OCC 충돌 메시지.

## 8. 리스크 / 마이그레이션

- **트리 규모(결정 1=a)**: 조직 메타 전체 로드. 자산 제외라 수천 행 수준 가정 — 과하면 (b) lazy로 후퇴(스펙 밖).
- **organizationStore 제거/축소**: TreeVisualization 등 다른 소비처가 있으면 함께 전환(전수 확인). reorder API(드래그)는 유지하되 staged화 여부는 이번 범위 밖(즉시 유지 가능 — 명시).
- **이중 floor 개념**: org `floors`(CRUD) vs 기존 `floor`(캔버스 설정) 섹션 병존 — 혼동 방지 주석.
- **부분 커밋 실패**: 단일 트랜잭션이라 원자적(기존 보장). 

## 9. 범위 밖
- 드래그 reorder의 staged화(현행 즉시 유지).
- 트리 lazy 로딩(1=a로 eager 채택).
- 조직 외 기존 "커밋 후처리 변전소-스코프" 잔재(B와 별개; 전역 커밋이라 대부분 해소되나 invalidate/archive 스코프는 별도 정리 대상).
