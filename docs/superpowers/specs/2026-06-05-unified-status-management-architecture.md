# 통합 현황관리 아키텍처 — 북극성 + 단계 로드맵

- 작성일: 2026-06-05
- 상태: 방향 승인됨 (사용자 동의). 각 단계는 별도 spec→plan→구현.
- 성격: 전략 아키텍처 문서. 구현 spec 아님 — 단계 분해 + 각 단계의 목표를 정의한다.

---

## 0. 문제 재정의

이 시스템은 **현황관리 시스템**이다. 배치도·실장도·전원계통도·네트워크 토폴로지를 그리는 *이유 자체*가 "무엇이 어디 있고, 무엇이 무엇과 연결되며, 전원이 어떻게 흐르고, 회선 경로가 어떤지"라는 **현황을 관리**하기 위함이다. 따라서 "도면 그리기"는 별도 활동이 아니라 **현황을 관리·표현하는 하나의 수단/뷰**여야 한다.

현재 시스템은 이를 "도면 그리는 플로우"와 "현황 보는 플로우" **두 도구로 분열**시켰다(진입점·멘탈모델·working copy·커밋 경로가 따로). 특히 **연결성(케이블·회선·광경로)이 도면 전용**이라, 현황의 핵심인 연결·계통이 "그리기 도구"에 갇혀 있다.

## 1. 업계 표준 (출처 검증 — deep-research)

우리 도메인과 가장 닮은 시스템(ETAP, Esri ArcGIS Utility Network, IQGeo, 변전소 OLND 학술문헌)의 공통 패턴:

1. **단일 연결성 모델(SSOT) + 다중 뷰 = 투영.** "CAD/그리기"와 "기록/대장"을 분리하지 않고, 하나의 자산+연결 모델 위에 지도·계통도·랙·표를 렌더링만 다르게.
2. **계통도(전원 one-line·토폴로지)는 손으로 안 그림 — 연결 모델에서 자동 생성·자동 배치**(규칙 + 그래프 드로잉).
3. **기하 뷰(축척 배치도) vs 위상 뷰(축척 없는 계통도) 이중 표현** — 계통도는 파생·재생성 가능한 비권위 투영.
4. **공유 선택 / 동적 투영** — 한 뷰 선택이 다른 뷰에 반영("select-once, see-everywhere").
5. **컨테인먼트 계층**(site→구조물→장비→디바이스→포트)으로 시각 밀도와 모델 분리.

> 핵심 논거(arXiv 1903.09495, 검증): 기록 DB와 별도로 손으로 그린 계통도를 따로 유지하면 자동 동기화가 안 돼 DB-뷰 불일치가 생긴다. → 통합해야 하는 이유.

출처: Esri Utility Network 문서, ETAP intelligent SLD/Auto-Build, IQGeo Network Manager, arXiv 1903.09495, IEEE one-line auto-layout. (전체: deep-research 리포트 `wfsfo282y`.)

## 2. 목표 아키텍처 (북극성)

**변전소 워크스페이스 = 단일 진입. 하나의 모델 위 여러 뷰를 전환하며 현황을 관리. 선택은 모든 뷰에서 공유. 계통도는 연결 모델에서 자동 생성.**

- **하나의 모델**: 이미 보유 — Asset 트리 + Cable 그래프 + FiberPath + DistributionCircuit(백엔드 SSOT).
- **뷰(모두 같은 모델의 투영)**: `표(register)` · `배치도(floor plan, 기하)` · `실장도(rack)` · `전원계통도(one-line, 자동)` · `네트워크 토폴로지(자동)` · `연결(connections)`.
- **공유 선택**: 선택된 자산이 워크스페이스 컨텍스트. 표에서 고르면 배치도·계통도에서 하이라이트, 역방향도.
- **연결성 1급**: 케이블/연결이 도면 전용이 아니라 모델의 1급 시민 — 표·패널에서 조회·편집, 그래프가 계통도를 구동.
- **계통도 자동생성**: 토폴로지(이미 cableTracer로 자동) + 전원계통도(DistributionCircuit/Cable 그래프 → 그래프 드로잉)를 손으로 그리지 않고 생성.

### 우리가 이미 가진 토대 (재활용)
- (나) 워크스페이스 셸 → **뷰 스위처**의 토대
- (가) 공유 패널·cross-nav → **공유 선택**의 토대
- Lv1 워킹카피 엔진 + OCC → **통합 스테이징**의 토대
- 자동 토폴로지(cableTracer) → **계통도 자동생성**의 증명

→ 재작성이 아니라 방향 전환. "두 도구" → "하나의 모델·여러 뷰·공유 선택·자동 계통도".

## 3. 단계 로드맵 (각 단계 독립 spec→plan→구현)

| 단계 | 내용 | 가치 | 비고 |
|---|---|---|---|
| **A. 뷰 스위처 + 공유 선택** | (나) 탭 → 뷰 스위처(표/배치도/실장도/…), 진입 일원화, **선택을 모든 뷰에서 공유**(지속 하이라이트) | "두 플로우 분열"을 직접 해소 — 최우선 | (나)+(가) 진화. 중간 규모 |
| **B. 연결성 1급화** | 변전소 단위 "연결" 뷰/표 — 무엇이 무엇과 연결되는지 캔버스 밖에서 조회·편집. 백엔드 connections-by-substation | 현황의 핵심을 그리기 도구에서 해방 | 큰 작업 |
| **C. 계통도 자동생성** | 토폴로지를 모달→1급 뷰로 승격 + **전원계통도(V4) 자동생성**(그래프 드로잉 auto-layout) | 사용자 강조점("계통도") | 가장 큼(알고리즘) |
| **D. 커밋 통합 (=5b)** | 두 커밋 경로(plan-save vs asset-commit)를 변전소 단위 한 스테이징으로 수렴 | 일관성·동시성 완결 | 토대 리팩토링, 비가시 |

**순서 근거:** A가 "분열 해소"를 가장 직접·저비용으로 달성하고 B/C/D의 골격(공유 선택 컨텍스트·단일 진입)을 제공하므로 **A 먼저**. 이후 사용자 강조점인 C(계통도)로 갈지, B(연결성 가시화)로 갈지는 A 후 재논의. D는 B/C 중 정당화되면 폴딩.

## 4. 미해결 / 원칙

- **좌표 출처**: 배치도는 수동 좌표(기하 진실) 유지, 계통도는 자동 배치(재생성). 표준의 geo-vs-schematic 분리를 따른다. 자동 배치는 "auto-layout이 모델-링크된 편집가능 위치를 시드"하는 하이브리드도 가능(후속 결정).
- **비권위 투영 원칙**: 계통도/토폴로지는 모델에서 파생 — 그 위 편집은 모델을 바꾸는 것이지 별도 산출물을 만드는 게 아니다.
- **범위 경고**: 전체는 2a급 이상 재구성. 반드시 단계로 쪼개 각자 shippable 하게.
- EAM/BIM(Maximo·Archibus)은 별도 CAD를 id로 링크만 하기도 함 — 자동생성 패턴은 우리 도메인(전력·네트워크)에서 가장 강함(우리에 적용 타당).
