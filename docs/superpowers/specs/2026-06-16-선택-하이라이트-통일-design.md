# 선택·하이라이트·동기화·뷰포트 통일 설계 (Single Selection SSOT)

**작성일:** 2026-06-16
**상태:** 설계 승인 대기 → 구현 계획

## 1. 목표

분산·파편화된 선택/하이라이트/동기화/뷰포트 로직을 **하나의 선택 SSOT → 하나의 해석기 → 순수 투영** 구조로 통일한다. 분기·특수케이스·트리거 개수를 최소화해 버그 표면을 구조적으로 없애고, 토폴로지 종료 시 평면도가 딤·이동된 채 남는 버그를 근본 해소한다.

**측정 기준(복잡도 감소):** 하이라이트 작성자 6→1, 뷰포트 effect 2→1, 카메라 트리거 다수→`focusTick` 단일, 한 요소 navy 강조 3겹→1겹, `onClickPlan`/`useCanvasEvents` 특수 clear 제거.

## 2. 현재 파편화 (진단 요약)

- **선택 SSOT 분산:** 자산은 `useSelectionStore(assetId/core/cableId)`, 캔버스 케이블은 `editorStore.selectedCableId` 별도.
- **하이라이트 소스 이중:** `useSelectionHighlight`(파생) + 6곳의 명령형 `startTrace`/`highlightDiagram`/`startCircuitTrace`.
- **렌더 3겹:** `renderEquipmentItems` 선택 styling + `useCanvas:113` 상세 글로우 + `ConnectionOverlay:180` 경로 글로우, 전부 `#15406b`, 한 자산에 누적.
- **카메라 결합:** `pathHighlight.active`/선택 변경이 뷰포트를 납치(`FloorPlanEditor` effect 2개).
- **특수 clear:** `useCanvasEvents`의 `clearHighlight` 3곳, `onClickPlan`의 진입 시 선택 비우기.

**이미 옳은 부분(유지):** 그리드↔사이드패널은 이미 `useSelectionStore.selectedCore`의 순수 단방향 투영(`FeederCircuitsPanel`/`SlotPortsPanel`/`ConnectionRegisterGrid`). 양방향 동기화 코드 없음.

## 3. 핵심 정체성: "회선/코어/포트/분기 = 케이블" 1:1

그리드의 회선/코어/포트, 사이드패널의 CB버튼/포트셀, 평면도의 경로는 모두 하나의 **케이블(연결)** 의 표현이다. 케이블 끝단은 `(assetId, core)` 주소로 1:1 식별된다. 따라서 **선택은 단 하나의 주소**이고 모든 뷰는 그 주소의 투영이다.

## 4. 타깃 아키텍처

```
useSelectionStore (SSOT)  { selectedAssetId, selectedCore, selectedCableId }
   │
   │ (모든 표면은 읽기만 — 순수 투영. 양방향 동기화 없음)
   │
   ├─ [직접 투영] 그리드 행      : row.(assetId,core) === (selectedAssetId,selectedCore) → 하이라이트
   ├─ [직접 투영] 사이드패널     : element.number === selectedCore → 활성·토글
   │
   └─ resolveSelection(assetId, core, cableId, graph)  ← 단 하나의 순수 해석기 (floor/연결 측)
        → { kind:'connection', cableId, nodeIds, placedIds, cableIds }
        → { kind:'asset', assetId }
        → { kind:'none' }
        │
        ├─ [파생 effect 1개] useSelectionHighlight → pathHighlightStore 캐시 작성 (유일 작성자)
        │     ├─ 평면도 캔버스   : kind==='connection' → 경로 딤/글로우
        │     ├─ 연결도 오버레이 : 케이블 강조
        │     └─ 연결도 카드     : card.cableId === resolved.cableId → 선택 표시
        │
        └─ [펄스] focusTick → 뷰포트 effect 1개 → resolveSelection 대상으로 포커스
```

**두 종류의 파생만 존재:**
1. **하이라이트(연속 투영):** 선택이 바뀌면 `useSelectionHighlight` 한 effect가 `resolveSelection`을 돌려 `pathHighlightStore`(파생 캐시)에 기록. 캔버스/오버레이/카드가 읽음.
2. **카메라(펄스):** `focusTick` 증가 시 뷰포트 effect 1개가 현재 선택을 `resolveSelection`해 그 대상으로 이동.

`mode` 플래그는 없다 — 딤/카메라 여부는 `resolveSelection().kind === 'connection'` 으로 결정된다.

## 5. 선택 SSOT 모델

`useSelectionStore` (변경 없음, 그대로 SSOT):
- `selectedAssetId: string | null` — 선택 자산(컨테이너/모듈/피더/슬롯/포트 등).
- `selectedCore: number | null` — 끝단 주소(광 core / CB number / `FEEDER_INPUT_CORE` 센티넬). 연결 식별의 핵심.
- `selectedCableId: string | null` — 정확 앵커(core-null 다중 컴포넌트 유일 구분, 케이블-우선 클릭 시 사용).

**선택을 쓰는 곳(트리거):** 그리드 행 클릭, 사이드패널 CB/포트 클릭, 캔버스 자산/케이블 클릭, 연결도 카드 클릭, 딥링크. 모두 `useSelectionStore` setter만 호출(`setSelected`/`setSelectedComponent`/`setSelectedCore`/`setSelectedAssetId`). **`pathHighlightStore`를 직접 호출하지 않는다.**

**캔버스 케이블 클릭:** `editorStore.selectedCableId`(웨이포인트 편집 UI 전용)는 유지하되, 동시에 SSOT에 연결 선택을 기록한다 — 케이블의 대표 끝단 `(assetId, core)`로 `setSelectedComponent(assetId, core, cableId)`. 그래야 그리드/패널/평면도가 함께 정렬된다.

## 6. `resolveSelection` — 단일 해석기 (순수 함수)

위치: `frontend/src/features/workspace/resolveSelection.ts` (기존 `selectionHighlight.ts`의 `resolveHighlight`/`resolveSelectedCable`와 `startCircuitTrace` 로직을 흡수·통합).

```ts
type Resolved =
  | { kind: 'connection'; cableId: string; nodeIds: Set<string>; placedIds: Set<string>; cableIds: Set<string> }
  | { kind: 'asset'; assetId: string }
  | { kind: 'none' };

function resolveSelection(
  assetId: string | null,
  core: number | null,
  anchorCableId: string | null,
  graph: TraceGraph,
  components: DiagramComponent[],
  effectiveAssets: Asset[],
): Resolved
```

**해석 순서(기존 규칙 보존):**
1. `assetId` 없음 → `none`.
2. `anchorCableId` 있음 → 그 시드/소속 컴포넌트 → `connection`(컴포넌트의 nodeIds/cableIds, placedIds는 `expandToPlacedIds`).
3. `(assetId, core)` → `resolveSelectedCable` → 케이블 → 소속 컴포넌트 → `connection`.
4. `core != null` 이고 `core`로 컴포넌트 매칭 → `connection`.
5. **회로 팬아웃**(선택 자산이 분기/피더 회로) → 그 회로에 물린 케이블 + 반대편 끝단 1-hop → `connection`(기존 `startCircuitTrace` 로직).
6. 위 어디에도 안 걸리면(빈 슬롯/단순 자산) → `asset`.

**역매핑(케이블 → 주소) 보장:** 케이블-우선 클릭이 `(assetId, core)`로 환원되도록, 컴포넌트/케이블에서 끝단 주소를 찾는 헬퍼를 같은 모듈에 둔다(그리드/패널 정렬용).

## 7. 하이라이트 파생 (유일 작성자)

`useSelectionHighlight` (워크스페이스 1회 마운트, 기존 위치):

```ts
useEffect(() => {
  const r = resolveSelection(selectedAssetId, selectedCore, selectedCableId, graph, components, effAssets);
  const hi = usePathHighlightStore.getState();
  if (r.kind === 'connection') hi.setHighlight(r);   // 파생 캐시 기록
  else hi.clearHighlight();                            // asset/none → 하이라이트 없음
}, [selectedAssetId, selectedCore, selectedCableId, graph, components]);
```

`pathHighlightStore`는 **파생 캐시**로 강등:
- 보유: `active`(=highlight 존재), `nodeIds`/`placedIds`/`cableIds`, `projection`(토폴로지용), `modalOpen`.
- 공개 API: `setHighlight(resolved)`(파생 effect 전용), `clearHighlight()`, `prepareTopology(cableId)`, `openTopology()`/`closeTopology()`.
- **제거:** 외부 호출용 `startTrace`/`highlightDiagram`/`startCircuitTrace`. (계산 로직은 `resolveSelection`으로 이동.)

## 8. 카메라 규칙 (분기 최소)

**카메라는 오직 `focusTick` 증가 시 1개 effect에서 이동한다.** 그 effect는 현재 선택을 `resolveSelection`해 대상으로 포커스:
- `kind==='connection'` → 경로 bounds(`cableIds`의 pathPoints)로 fit.
- `kind==='asset'` → 그 자산 rect로 center.
- `kind==='none'` → 이동 없음.

**`focusTick`을 bump하는 명시적 내비게이션(이것만):**
- 캔버스 더블클릭(자산 포커스).
- 딥링크 `?assetId=` 진입.
- **평면도 뷰 진입**(`useEditorSelectionBridge`, false→true 전이) — 현재 선택으로 포커스.
- **연결 선택**(그리드 행/패널/연결도 카드/캔버스 케이블 클릭) — "이 경로를 보여줘"는 명시적 동작이므로 bump.

**bump하지 않음:** 자산 단일 클릭(선택만), 토폴로지 모달 열기/닫기(카메라 중립), 기타 제자리 상태 변화.

→ `FloorPlanEditor`의 기존 viewport effect 2개(focusTick effect + pathHighlight-active effect)를 **1개로 합친다.**

## 9. 뷰 전환 시 선택 지속 규칙

- **같은 변전소, 다른 뷰** → 선택 **유지**. 각 뷰는 투영만. 평면도 진입 시 `focusTick` bump로 현재 선택에 포커스(자산/연결 무관 동일 규칙).
- **변전소/노드 변경** → 선택 **리셋**(선택 자산이 새 노드에 없음. `WorkspacePage` 노드변경 리셋 유지).
- **첫 컨테이너 자동 선택**(선번장/계통 진입) → **선택이 비어있을 때만** 발동(기존 사용자 선택을 덮지 않음).
- **`onClickPlan`의 진입 시 선택 비우기 특수 케이스 제거**(통일 모델로 흡수).

## 10. 렌더 dedup — 요소당 강조 1겹

한 요소(설비/케이블)는 **최대 하나의 강조**만 받는다. 우선순위:
1. **선택된 바로 그 요소** → 선택 styling(navy fill/stroke).
2. 아니고 **경로상 요소**(`placedIds`/`cableIds`) → 경로 styling(글로우 또는 정상 + 주변 딤).
3. 그 외 → 정상, 또는 `kind==='connection'`일 때 경로 밖이면 0.2 딤.

**구체 변경:**
- `useCanvas`의 상세 글로우(113-131줄)를 제거하고, 선택 요소는 `renderEquipmentItems`의 선택 styling 하나로 표현(navy 중복 제거). 선택 자산이 경로상에 있어도 선택 styling 우선(경로 글로우 미적용).
- `ConnectionOverlay`의 경로 설비 글로우(180-198줄)는 선택된 요소엔 적용하지 않음(중복 방지) — 경로의 *나머지* 요소에만.
- 0.2 딤은 `kind==='connection'`일 때만(경로 집중).

## 11. 토폴로지 모달 분리

`NetworkTopologyModal`은 `projection`+`modalOpen`만 읽으므로(`active`/edgeIds 미사용), 평면도 하이라이트/카메라와 분리한다.
- `prepareTopology(cableId)`: `loadProjection`으로 projection만 계산해 `set({ projection, modalOpen:true })`. **`active`/highlighted*/카메라 불변.**
- 토폴로지 진입점(`AssetConnectionsTab`/`AssetConnectionsSection`)은 `prepareTopology` 사용. 평면도는 그 시점 선택의 투영(연결이면 경로 강조) 그대로 유지.
- `closeTopology`: `set({ modalOpen:false })`만. 평면도는 선택 투영 그대로 → 종료 시 되돌릴 부작용 없음.

## 13. 테스트 전략

- **순수 단위 테스트(가능·우선):** `resolveSelection` — (asset,core)/anchorCableId/회로팬아웃/빈슬롯/none 각 분기. 케이블→주소 역매핑. 기존 `resolveHighlight` 회귀 케이스 이전.
- **빌드·기존 테스트:** `npm run build` 타입체크, `vitest run` 그린 유지.
- **브라우저 수동 검증(시각/카메라):**
  1. 연결 클릭(그리드/패널/연결도/캔버스케이블) → 평면도 경로 하이라이트 + 카메라 이동 + 패널 CB/포트 활성(1:1).
  2. 자산만 선택 → 선택 표시만, 경로/카메라 없음.
  3. 토폴로지 열고 X로 닫기 → 평면도 카메라·딤 불변(증상 회귀 없음).
  4. 뷰 전환(선번장↔평면도) → 선택 유지 + 진입 시 그 대상 포커스.
  5. 변전소 변경 → 선택 리셋.
  6. 한 요소에 navy 강조가 1겹만(겹침 없음).

## 14. 리스크·단계화

- **블래스트 반경:** 선택 진입점 다수 + 렌더러. 단, 각 변경이 "명령형→선택 setter", "중복 렌더 제거"로 국소적.
- **단계화 제안:** (P1) `resolveSelection` 추출+단위테스트 → (P2) 하이라이트 파생 단일화(명령형 제거) → (P3) 카메라 effect 통합 → (P4) 렌더 dedup → (P5) 토폴로지 분리 + 특수 clear 제거 → (P6) 뷰 전환 지속. 각 단계 후 빌드·수동검증.
- **보존:** 그리드↔패널 selectedCore 투영(이미 옳음), `editorStore.selectedCableId` 웨이포인트 편집 기능, 케이블 그리기(`interactionStore`)는 범위 밖.

## 15. 미해결/주의

- `resolveSelection`의 회로 팬아웃(5번)과 컴포넌트 매칭(2~4번) 우선순위가 한 케이스에서 겹치지 않는지 구현 시 확인(현 `startCircuitTrace`는 BRANCH asset 한정 → 분기 자산일 때만 5번 진입).
- 케이블→주소 역매핑이 다중 끝단(양 끝이 같은 변전소)일 때 어느 끝단을 선택으로 삼을지: 현재 보고 있는 자산 컨텍스트 우선, 없으면 source 끝단.
