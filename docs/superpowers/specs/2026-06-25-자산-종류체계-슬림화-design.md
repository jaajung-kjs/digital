# 자산 종류체계 슬림화 — code→id, isActive·미사용 필드 제거 설계

## 1. 배경 / 목표

`role` 단일화 이후에도 `AssetType`/카탈로그에는 **고정 자산체계 잔재**와 **한 번도 안 읽히는 필드**가 남아있다. 종류는 이제 사용자가 자산관리에서 직접 추가/편집하는데, 그 모델에 다음이 끼어 있다:
- `code`: 사용자 생성 시 backend가 `MOD-<랜덤uuid>`를 박는 의미 없는 두 번째 id. 의미 있는 code(RTU/OFD…)는 구 고정 taxonomy 잔재.
- `isContainer`(role과 1:1·미사용)·`defaultSlotSpan`(전부 1)·`requiredToCreate`(전부 `['name']`)·`iconName`(전부 null)·`displayColor`(ISA-101이 대체)·`fieldTemplate`(렌더 비활성·저장처 없음).
- `isActive`: 소프트삭제 플래그인데 **false로 만드는 곳이 0**, 전부 하드삭제 → 필터가 항상-참 no-op. (모든 모델 공통.)
- `CableGroup.kind`: 시드만 쓰고 읽는 로직 0.

**목표:** 종류 모델을 "이름만 사용자 입력"으로 슬림화. 분류는 `role`, 참조는 `id`(rename-stable PK). 위 필드 전부 제거.

## 2. AssetType 필드 정리

- **드롭:** `code` · `isContainer` · `defaultSlotSpan` · `requiredToCreate` · `iconName` · `displayColor` · `fieldTemplate` · `isActive`
- **유지:** `id` · `name` · `role` · `categoryId` · `sortOrder` · 노무필드(`laborType`/`installHoursPerUnit`/`removeHoursPerUnit`/`relocateHoursPerUnit`) · timestamps
- 부수: `CableGroup.kind` 드롭, `isActive` 전 모델 드롭(§4).

근거(증거): `isContainer`는 role과 1:1이며 로직 reader 0. `defaultSlotSpan`은 전 타입 1. `requiredToCreate`는 전 타입 `['name']`, 검증 reader 0. `iconName`은 전 타입 null. `displayColor`는 렌더가 ISA-101 무채색 토큰을 쓰고 "신뢰 안 함" 명시. `fieldTemplate`은 `AssetInspector`가 의도적으로 제외(자산별 값 저장처 없음 — `Asset.attributes` 제거됨).

## 3. `code` → `id` 참조 전환 (핵심)

`AssetType.code`(@unique) 컬럼 제거. 모든 참조를 `id`(PK)로. 이름은 사용자가 바꿀 수 있어 참조 키로 부적합 → `id` 사용.

### 3.1 시드 멱등 + 시드 내부 참조 — 이름 기반 (R2 패턴)
- 시드 종류는 각 행에 **로컬 키**(`key`: RACK/OFD/RTU… = 구 code 문자열 — DB 컬럼 아님)만 유지. 이전 `group→category` 로컬키 패턴과 동일.
- **멱등 키 = name**: `findFirst({ where: { name } })` → update/create (R2 케이블 카테고리와 동일). 기존 DB 행을 **이름으로 매칭**하므로 기존 random id·FK가 보존된다(결정적 id 재할당 불필요 — assets.assetTypeId 깨짐 회피). 전제: 시드 종류 26개의 `name`은 서로 유일(현 시드 확인됨).
- 시드 단계에서 `Map<key, id>`(upsert 결과 id)를 만들어 jikhal/preset 단계가 공유.

### 3.2 RackPreset.modules JSON
- `RackPresetModule`: `categoryCode: string` → `categoryId: string`.
- 검증(`rackPreset.service` ~113-123): `assetType.findMany({ where: { code: { in } } })` → `where: { id: { in } }`.
- **데이터 마이그레이션**: 기존 `rack_presets.modules` JSON의 각 모듈 `categoryCode` → 해당 AssetType.id 로 변환(드롭 마이그레이션 직전, code가 아직 있을 때 SQL/스크립트로 매핑). 변환 못 한 모듈은 로그 후 스킵(프리셋 자체는 유지).
- 시드 `rackPresets.ts`: 모듈을 로컬 키로 적고 `assetTypeId(key)`로 `categoryId` 해소해 JSON 저장.

### 3.3 jikhalAssets 시드
- `assets.json`의 `typeCode` → §3.1 의 공유 `Map<key, id>`(key = 구 code 문자열)로 해소. 현 `findMany({select:{id,code}})` map 빌드를 공유 맵으로 대체.

### 3.4 프론트
- `PresetActionsBar`/`FloorPlanEditor`: `codeToCategory`(code→cat) → `idToCategory`(id→cat); `mod.categoryCode` → `mod.categoryId`.
- `SaveRackAsPresetDialog`: `categoryCode: m.assetType.code` → `categoryId: m.assetType.id`(또는 `m.categoryId`).
- `assetPlanMapper.assetToRackModule`: `categoryCode` 필드 제거(`categoryId` 이미 있음).
- `rackModuleCategory.service` DTO + `RackModuleCategory` 타입: `code` 제거.
- 사용자 생성 종류: 랜덤 id, code/키 없음(시드 참조 대상 아님). `catalogCommit`의 `code: MOD-…` 생성 제거.

## 4. `isActive` 전면 제거

10개 모델에서 컬럼 드롭: `CableGroup`·`CableCategory`·`AssetCategory`·`AssetType`·`RackPreset`·`Headquarters`·`Branch`·`Substation`·`Floor`·`User`.

- **백엔드 필터(13곳)** `where:{ isActive:true }` 제거 → 무조건 조회: organizationTree(HQ/Branch/Sub/Floor)·assetCategory·assetType·branch(scoped)·headquarters·floor(scoped)·rackModuleCategory·cableGroup. `substation.service`의 옵션 `isActive` 필터 파라미터 제거.
- **auth.service**: 로그인(`if(!user.isActive) throw AccountDisabledError`) + 리프레시 토큰(`if(!storedToken.user.isActive)…`) 체크 2곳 제거. `AccountDisabledError`가 다른 곳에서 안 쓰이면 정의도 제거. → 계정 차단=하드삭제만(승인됨).
- **프론트**: 모든 DTO/타입에서 `isActive` 제거(`rackPreset`/`cableGroup`/`cableCategory`/`organization`/`asset`/`rackModule`/`assetCategory`/`substation`). 클라이언트 필터 제거: `PresetActionsBar`(p.isActive)·`EditorInsertBar`(presets/cableGroups)·`CableTypePicker`·`CableSpecModal`·`ConnectionLegend`·`CategoryComboboxPopover`. create 폼 `isActive:true` 기본값 제거(AssetTypesTab·CableTypesTab·assetCategory).
- **시드**: `isActive` 미기록.

## 5. 마이그레이션 (2단계 + 데이터 변환)

C5/role 단일화와 동일한 소비처-0 → 드롭 2단계. 단 preset JSON은 드롭 전에 데이터 변환 필요.

- **Phase A (소비처 0):**
  - 백엔드 reader/필터/DTO를 id·무필터로 전환(§3·§4), auth isActive 체크 제거, `catalogCommit` code 생성 제거.
  - 시드 결정적 id + 로컬 키 해소(§3.1~3.3), `isActive`/드롭필드 미기록.
  - 프론트 id 참조 + isActive/필드 제거(§3.4·§4).
  - **preset JSON 데이터 변환**: 기존 `rack_presets.modules` categoryCode→categoryId(code 살아있을 때).
  - 게이트: `code`/`isActive`/드롭필드 소비처 0(grep) + tsc 0 + vitest 전수 + fresh seed + 스모크.
- **Phase B (드롭):** schema에서 `code`·`isActive`(10모델)·`isContainer`·`defaultSlotSpan`·`requiredToCreate`·`iconName`·`displayColor`·`fieldTemplate`·`CableGroup.kind` 제거 + 마이그레이션 + prisma generate.

## 6. 비목표

`role` enum 값 불변. `AssetCategory`/`CableGroup`(고정5)/`CableCategory` 구조 불변. `Cable.categoryId` 불변. 노무필드 불변. 새 헬퍼/상수/role-서브셋 신설 금지(리터럴 인라인).

## 7. 리스크 / 결정

- **preset JSON 데이터 마이그레이션**: 기존 행 code→id 변환 필수(누락 시 프리셋 모듈 깨짐). 변환은 code가 살아있는 Phase A에서.
- **User.isActive 제거 = 인증 가드 제거**(승인됨): 계정 비활성화 개념 소멸, 차단=하드삭제.
- **비가역 드롭**: Phase A 소비처 0 검증이 게이트.
- **시드 이름 유일성**: name 기반 멱등은 시드 종류 name이 유일해야 함(현 26개 확인됨). 새 시드 종류 추가 시 유일 유지 — 깨지면 findFirst가 오매칭.

## 8. 구현 단계 (plan에서 분할)

- **S1 백엔드 code→id**: rackPreset 검증/저장·assetPlanMapper·rackModuleCategory DTO·jikhal/assetTypes/rackPresets 시드(name 기반 멱등 + 공유 key→id 맵)·catalogCommit code 제거.
- **S2 백엔드 isActive 제거**: 13 필터 + substation 옵션 + auth 체크 2곳 + AccountDisabledError + 시드.
- **S3 프론트**: code→id 참조(preset/save dialog) + isActive DTO/필터 제거 + AssetType 드롭필드 타입 정리.
- **S4 preset JSON 데이터 마이그레이션** (code→id, code 살아있을 때).
- **[CHECKPOINT] 소비처 0.**
- **S5 Phase B 드롭** (code·isActive·6필드·CableGroup.kind 컬럼 드롭 + 마이그레이션).
- **S6 최종 회귀 + fresh seed + 스모크.**
