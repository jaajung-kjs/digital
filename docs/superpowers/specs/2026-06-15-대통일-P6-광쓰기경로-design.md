# 연결 모델 대통일 — P6 광 쓰기 경로 마이그레이션 (클린 커오버) Design

**작성일:** 2026-06-15
**상태:** 설계 확정(사용자 승인) → 구현 플랜 대기

## 배경

연결 모델 대통일의 읽기(VIEW) 측은 완료됐다(P3a 선번장, P3b 하이라이트·토폴로지가 슬롯/OUT/OPGW 신모델 위 단일 `cableTrace`로 동작). 그러나 **광 연결을 생성/편집하는 쓰기 경로는 아직 구 `FiberPath` 모델**이다:

- **OfdPortPicker** (평면도 케이블 드로잉): 설비→OFD 케이블을 그릴 때 FiberPath의 포트를 골라 `Cable.fiberPathId`/`fiberPortNumber`로 저장.
- **FiberPathManager** (OFD 자산 상세패널): FiberPath(OFD↔OFD 경로 + portCount) 생성/삭제.
- **ConnectionDiagram** (자산 상세 광 연결 표시): `composeFiberPaths`/`useEffectiveFiberPaths`로 구모델 읽기.
- 백엔드 `FiberPath`/`FiberCore` CRUD + `Cable.fiberPathId`/`fiberPortNumber` 컬럼.

**문제:** 지금 그린 광케이블은 구모델로 가서 신 선번장에 안 보인다(데이터 분기). 또 P4 마이그레이션이 **추가형**이라 같은 물리 연결이 두 번 표현돼 있다 — 옛 광케이블(설비↔OFD, `fiberPathId`)과 새 OUT 케이블(설비↔슬롯, `number`). 슬롯은 미배치라 `floorAnchor`가 부모 OFD로 앵커링하므로 **평면도에 중복 선**으로 렌더된다.

**목표:** 쓰기 경로를 신모델(슬롯/OPGW/OUT)로 전환하고, P4가 만든 옛 중복 표현을 정밀 제거해 **신모델을 유일 진실로** 만든다(클린 커오버). 백엔드 `FiberPath`/`FiberCore` 모델·스키마·컬럼의 물리적 삭제는 프론트 무참조 확인 후 **P7**로 분리(저위험).

## 신모델 쓰기 동작 매핑

| 동작 | 구 모델 | 신 모델 |
|---|---|---|
| 경로 생성 (OFD↔OFD) | FiberPath 2개(양방향) + portCount | **슬롯 2개**(각 OFD 자식 `OFD-SLOT`, connectionKind `conduit`, 이름=대국명) + **OPGW 케이블 1개**(IN-IN, `specParams.cores=N`) — 단일 양방향 객체 |
| 코어 연결 (설비↔코어) | 케이블 + `fiberPathId`/`fiberPortNumber` | **OUT 케이블**(설비↔슬롯, slot 끝 `role=OUT`, `Cable.number=코어#`, 메타=`specParams`) |
| 빈 코어 | FiberCore/포트 레코드 | 암묵적 — OPGW `specParams.cores`가 용량, OUT 케이블 없으면 빈 코어. **미리 생성하지 않음.** |

핵심 불변식(P3a/P4와 동일): 슬롯=conduit 자식(부모=OFD), OPGW=슬롯↔슬롯 IN-IN, OUT=설비↔슬롯 number 코어. 한 경로당 슬롯 2개·OPGW 1개. `cableTrace`가 conduit 채널짝으로 코어를 끝까지 추적.

## 아키텍처 / 컴포넌트

### A. 경로 생성 — OFD 상세패널 경로 관리자 (FiberPathManager 재작성)

- **위치:** OFD 자산 상세패널(기존 `OfdPathsView`/`FiberPathManager` 자리).
- **목록:** 그 OFD의 슬롯들(=경로). 각 행 = 슬롯의 대국 변전소명(`remoteSlotSubstation` 투영) + 코어 용량 + 사용 수.
- **경로 추가:** 대국 OFD 선택(전역 자산 목록에서 OFD code 필터, cross-substation) + 코어 수 입력(기본값 예: 24) → 워킹카피에 **슬롯 2개**(`put('assets', …)` — 각 OFD 자식, assetTypeId=OFD-SLOT, substationId=각 부모 OFD의 변전소) + **OPGW 케이블**(`put('cables', …)` — 두 슬롯 IN-IN, cableType FIBER, category CBL-OPGW, `specParams.cores=N`) 스테이징. 전역 워킹카피·전역 커밋이라 양 변전소 객체를 한 커밋에 생성.
- **경로 삭제:** 슬롯 2개 + OPGW + 그 슬롯에 달린 OUT 케이블 제거(`remove`). (cascade는 프론트에서 명시적 — 슬롯 제거 시 OUT 고아 방지.)
- **"상대 대국 자동설치"는 공짜:** OPGW 단일 객체가 양쪽 슬롯을 잇는다 — 역방향 중복 없음(구모델의 reciprocal FiberPath 문제 소멸).

### B. 코어 연결 — 평면도 드로잉 + 피커 (OfdPortPicker 재작성)

- **제스처:** 기존과 동일 — 평면도에서 설비→OFD 케이블 드로잉.
- **피커(SlotCorePicker, OfdPortPicker 재작성):** 대상 OFD의 슬롯들(경로) 나열 → **경로(대국) 선택** → **코어 번호 선택**(다음 빈 번호 자동제안, 점유 번호 표시, 수정 가능). 확정 시 **OUT 케이블** 스테이징(`put('cables', …)`: 설비↔슬롯, slot 끝 `role=OUT`, `number=코어#`, cableType FIBER, category=광케이블 category(기존 `CBL-*` 광, 드로잉 시 선택 — P4 OUT 케이블과 동일 형식)).
- **데이터 끝점=슬롯, 시각 렌더=OFD:** OUT 케이블의 끝점은 슬롯(미배치)이지만 `floorAnchor(slot)`가 부모 OFD로 해소 → 평면도에 설비↔OFD 선으로 렌더(기존과 시각 동일). 캔버스 드로잉 핸들러는 드롭 대상이 OFD면 슬롯으로 endpoint 치환.
- **로컬 코어만 생성:** 대국 설비 연결은 대국 운영자가 자기 슬롯에 OUT#동일번호로 별도 생성. 경로의 대국 슬롯은 이미 존재(A에서 생성). 선번장 상대국측은 대국 OUT이 있으면 `cableTrace` 투영으로 표시.
- **경로 부재 시:** 대상 OFD에 슬롯(경로)이 없으면 피커가 "경로 없음 — OFD 상세에서 경로 추가" 안내(인라인 생성은 비목표, A로 유도).

### C. 클린 커오버 — 중복 제거 + 신모델 읽기 전환

1. **옛 광케이블 정밀 삭제:** P4가 각 OUT 케이블 `specParams`에 심은 `__fromCableId`(출처 옛 광케이블 id) 마커로 매칭. 마이그레이션된 OUT 케이블 전체를 훑어 `__fromCableId` 집합을 만들고, 그 id의 **옛 fiber 케이블을 삭제**. 옛 **FiberPath 행도 삭제**(슬롯/OPGW가 대체). → 평면도 중복 선 해소.
   - 실행 방식: 일회성 backend 정리 스크립트(P4의 `migrateFiberToSlots.ts`와 동형 — idempotent, `__fromCableId` 기준, dry-run/revert 가능). 옛 fiber 케이블·FiberPath만 삭제, OFD-SLOT/OPGW/OUT(신모델)·FiberCore는 보존(FiberCore는 P7에서).
2. **ConnectionDiagram 전환:** 자산 상세 광 연결 표시를 `composeFiberPaths`/`useEffectiveFiberPaths`(구모델) → 신모델(슬롯의 OUT 케이블 + `traceGraph`/`remoteSlotSubstation` 투영)로.
3. **죽은 프론트 헬퍼 삭제(커오버로 무참조화):** `usePortStatus`·`overlayLocalStagedCables`·`useFiberPaths`·`fetchAllFiberPathsCached`·`composeFiberPaths`·`useEffectiveFiberPaths`·구 `OfdPortPicker`·구 `FiberPathManager`. (백엔드 `/fiber-paths` 호출이 사라지므로 그 라우트도 P7 후보가 되나, 라우트 자체 삭제는 P7.)

### D. P7로 남기는 것 (P6 비목표)

- 백엔드 `FiberPath`/`FiberCore` Prisma 모델·`fiberPath.service`·`fiberPaths.routes`·컨트롤러·커밋 스키마의 `fiberPaths`/`fiberCores` 섹션 삭제.
- `Cable.fiberPathId`/`Cable.fiberPortNumber` 컬럼 삭제(+ `cable.service`의 `buildFiberPathLabel`/`assertOfdFiberPath`, `substationCommit`의 해당 처리, 프론트 타입의 fiber 필드).
- 옛 `fiber_paths`/`fiber_cores` 테이블 drop + 잔존 데이터.
- P6에서 무참조가 된 백엔드 fiber 라우트/서비스 정리.

## 데이터 흐름

```
[경로 추가]  OFD 패널 → put(슬롯A, 슬롯B, OPGW) → 전역 overlay → POST /commit → 슬롯/OPGW 영속
[코어 연결]  평면도 설비→OFD 드로잉 → SlotCorePicker(경로+코어#) → put(OUT 케이블) → overlay → commit
[표시]       선번장/ConnectionDiagram/하이라이트/토폴로지 = traceGraph + cableTrace 투영(신모델, 이미 P3a/P3b)
[커오버]     일회성 스크립트: __fromCableId 매칭 → 옛 fiber 케이블·FiberPath 삭제(신모델만 잔존)
```

## 엣지/에러 처리

- **경로 삭제 시 OUT 고아:** 슬롯 제거 전 그 슬롯의 OUT 케이블도 함께 remove(프론트 cascade). 확인 다이얼로그에 "이 경로의 코어 연결 N개도 삭제" 경고.
- **코어 번호 충돌:** 같은 슬롯에 같은 number OUT 2개 방지(피커가 점유 번호 비활성/경고). 커밋 시 백엔드 추가 검증은 비목표(신모델은 number UNIQUE 제약 없음 — 피커 UX로 방지, 필요 시 P7에서 제약).
- **커오버 스크립트 안전:** dry-run + revert, `__fromCableId` 없는 옛 케이블(마이그레이션 안 된 것)은 건드리지 않음(보존). idempotent(이미 삭제된 건 skip).
- **cross-substation 경로:** 대국 OFD가 타 변전소 → 슬롯 substationId는 각 부모 OFD 변전소. 전역 커밋이 양쪽 원자 처리.
- **드로잉 대상이 OFD 아님:** 기존 비-OFD 케이블 드로잉은 무변경(피커는 OFD 드롭 시에만).

## 테스트 전략

- **순수 단위:** 경로 추가 페이로드 빌더(슬롯2+OPGW spec), 코어 연결 페이로드 빌더(OUT 케이블 number/role), 다음 빈 코어번호 계산, 경로 삭제 cascade 집합. 커오버 스크립트의 `__fromCableId` 매칭(순수 `planFiberCutover`).
- **백엔드:** 커오버 스크립트 dry-run/실행/revert + idempotency(psql 검증: 옛 fiber 케이블·FiberPath 0, 신모델·FiberCore 보존). tsc + 기존 테스트 회귀 0.
- **프론트:** build + tsc + 기존 테스트 그린(죽은 헬퍼 삭제로 관련 테스트 정리). 신 SlotCorePicker/경로관리자 단위.
- **수동 스모크(병합 전):** 경로 추가 → 평면도 설비→OFD 드로잉 → 피커로 코어 연결 → 선번장에 즉시 표시 → 평면도 중복 선 없음 확인.

## 비목표 (YAGNI)

- 인라인 경로 생성(드로잉 피커 내) — 경로 관리자로 일원화.
- 대국 설비 자동 연결 — 로컬 코어만.
- 엑셀 import, 코어 number DB UNIQUE 제약(피커 UX로 방지), 백엔드 모델/스키마 삭제(P7).

## 단계 분해(플랜 예고)

1. 경로 관리자(슬롯+OPGW 생성/삭제) — 페이로드 빌더 + OFD 패널 재작성.
2. SlotCorePicker(코어 연결 드로잉) — OfdPortPicker 재작성 + 캔버스 endpoint 치환.
3. ConnectionDiagram 신모델 전환.
4. 클린 커오버 스크립트(`planFiberCutover` + 실행/revert) — 옛 fiber 케이블·FiberPath 삭제.
5. 죽은 프론트 헬퍼 삭제 + 검증 + 최종 리뷰.
