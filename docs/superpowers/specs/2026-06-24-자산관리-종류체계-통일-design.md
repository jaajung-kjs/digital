# 자산관리 — 종류 체계 통일 설계 (Asset/Cable Type Catalog)

**작성일:** 2026-06-24
**상태:** 설계 확정 (구현 plan 대기)

---

## 1. 목표 (Goal)

자산종류·케이블종류·랙 프리셋을 사용자가 직접 CRUD 할 수 있는 **전역 "자산관리"** 진입점을 만들고,
그 전제로 **흩어진 분류 체계를 단일 `role` 축으로 통일**한다.
사용자는 **이름 + 분류**만 정의하고, 구조적 거동은 시스템이 `role`로 고정 소유한다.

> 본부 전체가 공유하는 표준 마스터 데이터(종류/프리셋)를 관리하는 기능. 자산 *인스턴스* CRUD가 아니다.

---

## 2. 아키텍처 (Architecture)

- **2축 분리:** 분류(taxonomy, 사용자 편집) vs 구조 역할(`role`, 시스템 고정). 현재 이 둘이 `code`/`group`/`placementKind`/`connectionKind`/`EquipmentKind` 4~5군데에 중복·분산 → `role` 하나로 통합.
- **A안 저장 모델:** 자산관리 화면은 **변전소 워킹카피와 분리된 자체 드래프트 + 전용 저장/취소**. 전역 마스터 편집과 변전소 작업·권한을 섞지 않는다.
- **표준 진입점 통일:** 모든 케이블 생성을 표준 케이블 진입점(`startCableConnection` + 종류 피커) 하나로 모아 케이블의 `role`/`code`를 제거한다.

## 4. 현재 문제 (Background)

구조적 역할이 다음 4~5곳에 중복·분산되어 드리프트의 원인이 됨:

| 위치 | 현재 역할 |
|---|---|
| `AssetType.code` | OFD/DIST/RACK/FEEDER/OFD-SLOT 특수 거동 + 케이블타입 파생 + 프리셋 참조 |
| `AssetType.group` | 공유 카테고리(분류) — 그런데 code와 의미 겹침 |
| `AssetType.placementKind` | RACK/OFD/DIST/GROUNDING/HVAC (배치설비 식별) |
| `AssetType.connectionKind` | distributor(피더)/conduit(슬롯) |
| `EquipmentKind`(프론트) | RACK/OFD/DISTRIBUTION/GROUNDING/HVAC — detail panel·캔버스 크기 구동 (placementKind와 사실상 중복) |

또한 케이블은 `CableCategory.code`('CBL-OPGW' 등)에 OPGW/광점퍼 자동생성이 하드코딩되어 사용자가 종류를 자유 편집하기 어려움.

## 5. 확정 데이터 모델 (Final Model)

### 5.1 자산종류 (AssetType)

```
AssetType = name(사용자) + categoryId(AssetCategory FK, 사용자) + role(시스템 enum, 숨김·불변)
```

**`role` 7값 (시스템이 코드/시드로 고정 정의, 사용자 선택 불가):**

| role | 의미 | 공간섹션(정보탭 주입) | 컨테이너 |
|---|---|---|---|
| `rack` | 랙 (도면 배치) | 내부 설비(실장도) `RackInternal` | ✅ |
| `ofd` | 광단국 (도면 배치) | 경로 `OfdPathsView` | ✅ |
| `panel` | 분전반 (도면 배치) | 회로 `DistributionCircuits` | ✅ |
| `slot` | 경로슬롯 (OFD 내부) | 포트 `SlotPortsPanel` | — |
| `feeder` | 피더 (분전반 내부) | 분기 `FeederCircuitsPanel` | — |
| `standalone` | 도면 단일 설치 (접지·공조 …) | 없음 (인스펙터만) | — |
| `device` | 랙 내부 모듈 / 일반 (**사용자 신규 기본값**) | 없음 (인스펙터만) | — |

규칙:
- **role당 여러 타입 허용** (rack 변형 여럿, standalone에 접지·공조 2종 등). 거동은 role이 구동하므로 같은 role의 타입은 동일하게 동작.
- role 보유 타입(rack/ofd/panel/slot/feeder/standalone)은 **시드로 정의 + 삭제 불가 + 이름만 수정**.
- **사용자가 새로 만드는 타입은 항상 `role = device`** (role 노출/선택 없음).
- `role`이 **공간섹션·컨테인먼트·트레이스**를 단일 구동 — `placementKind`·`connectionKind`·`EquipmentKind`·특수 `code`를 모두 대체.

**제거 필드:** `code` · `group`(→ AssetCategory 테이블) · `isContainer`(→ role 파생) · `placementKind` · `connectionKind` · `defaultSlotSpan` · `displayColor` · `fieldTemplate`. (`requiredToCreate`·`iconName`·`sortOrder`·`isActive`는 사용 여부 확인 후 결정 — 7.1 참조.)

### 5.2 케이블종류 (CableCategory)

```
CableCategory = name(사용자 자유) + displayGroup(고정 enum 5종: 광|전원|네트워크|제어|접지)
```

- **`displayGroup`은 자유 텍스트가 아니라 시스템 고정 enum** — 사용자는 5종 중 **선택**(타이핑 불가). 이것이 케이블의 **구조 축**(자산 `role`의 등가물)이며 도메인이 의존한다: 계통은 `displayGroup === '전원'`, 선번장/트레이스는 `cableType`. 케이블엔 자산처럼 별도 자유 분류가 없다 — 고르는 분류가 곧 고정 구조 enum.
- 케이블의 구조 타입(`Cable.cableType`: AC/DC/LAN/FIBER/GROUND)은 **`displayGroup`에서 결정적 파생** (현재 `code`→`MATERIAL_TO_CABLE_TYPE` 대체). 정준 매핑: 광→FIBER, 전원→AC, 네트워크→LAN, 제어→DC, 접지→GROUND. **단 현재 데이터에서 group↔cableType 1:1 여부(제어에 DC/LAN 혼재 등)는 P3에서 확정.**
- 백엔드 `cableCategory` create/update는 `displayGroup`을 이 5종 enum으로 **검증**(자유 텍스트 거부).
- **OPGW/광점퍼의 특수성은 카테고리가 아니라 연결 구조에서 옴:** `isOpgwTwin = cableType==='FIBER' && sourceRole==='IN' && targetRole==='IN'`. 트레이스/토폴로지는 이미 구조로만 판별(카테고리 무관).
- **케이블은 `role`·`code` 없음.** (제거: `code` · `displayColor` · `iconName` · `specTemplate`. `unit`·`description`는 7.1 확인 후.)

#### specParams vs specTemplate (놓치기 쉬운 부분)
- **`Cable.specParams` = 인스턴스 데이터, 그대로 보존. 종류 정의에는 params 스키마가 없다.**
  - 어떤 params가 붙을지는 **구조적 역할(IN/OUT) × 도메인(선번장/계통)**이 결정 — 종류 이름과 무관:
    - 광: IN-IN 트렁크 → `{cores, coreMeta}` (경로 생성·선번장) / OUT 점퍼 → `{}` (`buildCoreOutCable`)
    - 전원: 피더 IN 입력 → `{voltage}` (버스, voltageBus 규약) / 피더 OUT 분기 → `{capacity, switchState, number}` (`buildPowerRows`)
  - 선번장/계통 그리드가 **구조적 역할로** 케이블을 식별해 필드를 렌더(카테고리 code 안 봄) → code 제거에 영향 없음. **변경 없음.**
  - 따라서 사용자가 케이블 종류(이름+분류)를 자유 편집해도 안전 — params는 종류가 아니라 연결 구조에 붙는다.
- **`CableCategory.specTemplate` = 드롭.** 근거: ① `.params`(카테고리별 폼 스키마)는 **소비처 0개** — 그리드가 옵션을 하드코딩(`VOLTAGE_OPTIONS`/ON·OFF/`AmpField`). ② `.format`은 백엔드 `buildCableSpecification`의 `specification` 문자열 생성에만 쓰이나, 프론트는 그 대신 **`category.name`을 specification으로 사용**(`CableInspector`는 규격 미노출 명시). → `specification`을 `category.name`으로 대체.
- **P1 검증 게이트:** 시공 보고서(`constructionReport`)가 백엔드 `specification`(포맷 문자열)을 표시하는지 확인. 표시한다면 `category.name`으로 바뀌어도 수용 가능한지 결정 후 `specTemplate`·`buildCableSpecification` 드롭.

### 5.3 분류 테이블 (사용자 편집 taxonomy)

- **신설 `AssetCategory`** = `{ id, name, sortOrder, isActive }`. 자산종류 분류(예: 광전송장치).
- 케이블 분류는 기존 `CableCategory.displayGroup`을 그대로 활용(별도 테이블 불필요 — group이 곧 구조타입이자 분류).

> 색·아이콘 등 표시 속성은 DB가 아니라 **프론트에서 role/group 기준으로 관리**.

## 6. 케이블 생성 통일 (role 제거의 핵심)

모든 케이블 생성을 **표준 진입점(`startCableConnection` + 종류 피커)** 하나로 모은다.

### 6.1 선번장 OPJ 제거 (B안)
- `EquipmentSelectCell`(자국/대국 설비 드롭다운)의 **드롭다운만 제거 → 읽기전용 설비명 표시로 환원**. **컬럼 자체는 유지.**
- 코어↔설비 연결 생성은 이미 존재하는 표준 경로 사용: `SlotPortsPanel`의 "케이블 연결" → `startCableConnection({ source:{ slotId, coreNumber, role:'OUT' } })`. **신규 진입점 불필요.**
- 하드코딩 `categories.find(c => c.code === 'CBL-OPJ')` 제거.

### 6.2 경로슬롯 OPGW 통일
- 경로 생성 시 슬롯 2개(자국/대국)가 만들어지므로 **source/target은 그 2슬롯으로 고정** → 사용자는 **종류만 선택**(`OfdRoutePopover`에 종류 피커 추가).
- IN-IN 트윈 구조는 `buildRouteCreate`가 그대로 박고, 카테고리만 선택값으로 주입.
- 하드코딩 `categories.find(c => c.code === 'CBL-OPGW')` 제거.

### 6.3 케이블 편집 진입점 (P3)
케이블 편집은 세 진입점으로 분리된다:
- **생성:** 표준 진입점(`startCableConnection` + 종류 피커) — 종류+이름 선택 + 끝점.
- **종류(group)+이름 변경:** `CableInspector`(연결탭) — **구조 종류를 바꾸는 유일 진입점.** 현재 "전체 카테고리 평면 목록"을 **2단 피커로 교체**:
  - `종류`(group, 고정 enum 5종) 드롭다운 → `이름`(해당 group의 CableCategory) 드롭다운. 표시 "전원 - {이름}".
  - 이름 선택 시 `categoryId` 패치 + `cableType`을 새 group에서 재파생(생성시 고정 → 변경 추종으로 변경).
- **이름 변경(그룹 내):** 계통 `SpecCell` — 같은 group 안에서 다른 이름 선택(이미 group-scoped).
- **`Cable.label` 필드 완전 제거:** 불필요. 표시는 카테고리 `이름`으로 일원화.
  - `ConnectionOverlay`(`cable.label || categoryName` → `categoryName`), `cableToLocal`, `CableInspector` 라벨 행, 백엔드 `cable.service` DTO, 스키마 `Cable.label` 모두 제거.
- 정리 범위: 패치에서 `categoryCode` 제거(→ `categoryId`만), `displayColor`는 group/프론트 파생으로.

## 8. 자산관리 UI (A안)

- **진입:** 헤더 전역 메뉴 "자산관리", **관리자 전용**(`useIsAdmin()`), 별도 라우트(`/asset-management` 등). 탭 구조(현황/평면도/…)에 **포함하지 않음**(전역 스코프).
- **저장 모델:** 화면 자체 드래프트 + 전용 저장/취소(변전소 워킹카피와 분리). 저장 시에만 서버 반영. 변전소 작업 중 종류를 바꿔도 즉시 반영되지 않음(저장 후 반영).
- **탭 3개:**
  1. **설비종류** — 목록 + 추가/이름·분류 편집/삭제. role 보유 타입은 이름만 수정·삭제불가(잠금 배지). 신규=device.
  2. **케이블종류** — 목록 + 추가/이름·분류(group) 편집/삭제. FK(사용 중) 삭제 가드.
  3. **프리셋(랙)** — **랙 사이드패널 UI 재사용**. 저장은 **드래프트+저장으로 통일**(직접 서버 변이 제거).
- **공용 컴포넌트 강제:** `DetailCard`·`SectionItem`·`FormRow`·`EditableField`·`Badge`·`Button`. CONVENTIONS.md 준수.

- 자산/케이블 *인스턴스* 생성·편집 흐름(평면도·랙·선번장·계통) — 동작 불변.

## 11. 테스트 전략

- **단위:** role 파생 매핑(마이그레이션), `assetClassify` role 판정, `resolveSpatialSection` role switch, 케이블타입 group 파생, FK 삭제 가드.
