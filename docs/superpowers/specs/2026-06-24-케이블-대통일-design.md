# 케이블 대통일 — 사용자 정의 그룹·이름, cableType 제거 설계

**작성일:** 2026-06-24
**상태:** 설계 확정 (구현 plan 대기)
**선행:** [자산관리 종류체계 통일](2026-06-24-자산관리-종류체계-통일-design.md) (자산측 role 단일화 = P2 완료). 본 문서는 케이블측을 동일 철학(전부 사용자 정의 + 시스템 enum 제거)으로 통일한다.

---

## 1. 목표

케이블 종류를 **100% 사용자 정의**로 만든다 — 사용자가 **그룹(이름+색)**을 만들고 그 안에 **이름들**을 추가해 케이블 종류를 정의한다. 시스템이 박아둔 `cableType` enum(AC/DC/LAN/FIBER/GROUND)·`code`·하드코딩 색/그룹 매핑을 **전부 제거**한다. 트레이스의 "광" 의존은 **사용자 그룹 + 연결 구조**로 대체한다.

> 자산측과 대칭: 자산 = `이름 + 분류(AssetCategory) + role(시스템)`. 케이블 = `이름 + 그룹(CableGroup, 사용자)` — 케이블엔 시스템 enum 자체가 없다(구조가 그 역할).

---

## 2. 핵심 원리

- **케이블 종류 = 이름 + 그룹.** 그룹 = 이름 + 색(둘 다 사용자). 케이블 인스턴스는 종류(`categoryId`)를 가리키고, 표시 이름 = 종류 이름, 색 = 그룹 색.
- **`cableType` enum 제거.** "같은 종류 따라가기"(추적)는 **같은 그룹**으로, "광 트렁크(OPGW) 판별"은 **연결 구조(슬롯↔슬롯 IN-IN)**로.
- **색은 그룹 한 곳에서만.** 종류별/인스턴스별 색 오버라이드 없음.
- **`specParams`는 인스턴스 도메인 데이터로 불변** — 코어수/coreMeta(광), voltage/capacity/switchState(전원). 선번장/계통이 **구조적 역할(IN/OUT)** 기준으로 소유·편집(종류와 무관). 본 통일과 충돌 없음.

---

## 3. 데이터 모델

### 3.1 신설 `CableGroup` (사용자 CRUD)
```
CableGroup = { id, name(unique), color(VarChar7), sortOrder, isActive, createdAt, updatedAt }
```
예: `{ name: '전원케이블', color: '#ef4444' }`.

### 3.2 `CableCategory` 축소 — "케이블 이름"
```
CableCategory = { id, name, groupId(FK→CableGroup), sortOrder, isActive, timestamps }
제거: code · displayGroup(→groupId) · displayColor(→그룹) · specTemplate · unit · description · iconName
```
예: `{ name: 'Fr-sq3.5', groupId: <전원케이블> }`.

### 3.3 `Cable`(인스턴스) 정리
```
제거 컬럼: cableType · color(인스턴스 색 오버라이드) · label
제거 enum: CableType (AC/DC/LAN/FIBER/GROUND)
유지: categoryId(→종류) · specParams · sourceRole/targetRole · number · pathPoints/length 등
표시: 이름 = category.name, 색 = category.group.color
```

## 4. 트레이스 — `cableType` 의존 제거

| 현재 (cableType) | 대체 |
|---|---|
| 같은 종류 경로 추적 (`cableTrace`/`traceProjection`/`internalPath`/`traceRemoteEndpoints` 의 `c.cableType === seed.cableType`) | **같은 그룹**: `c.groupId === seed.groupId` (seed 케이블의 그룹). 기본값 'FIBER' 폴백은 seed 케이블 그룹으로. |
| OPGW 판별 (`isOpgwTwin`: `cableType==='FIBER' && IN-IN`) | **구조만**: `sourceRole==='IN' && targetRole==='IN'` (슬롯↔슬롯). IN-IN 은 OPGW 에만 나타남(전원 피더 IN 은 OUT-IN). |
| 슬롯 광케이블 필터 (`slotPorts`/`slotRegister`/`EquipmentSelectCell`/`OfdSlotRail`/`fiberWrite` 의 `cableType==='FIBER'`) | "슬롯에 닿는 케이블" 구조 기준(슬롯은 광 전용). |
| 색 폴백 (`CABLE_COLORS[cableType]`) | 그룹 색. |
| 미분류 폴백 (`CABLE_TYPE_DISPLAY_GROUP`) | 제거(categoryId 항상 존재). |

→ 트레이스가 `cableType`을 전혀 읽지 않음. 그래프에 `groupId`(케이블별) 동반 필요(현재 categoryId→group 해소).

---

## 5. 생성·편집 진입점

종류 선택 = 어디서나 **그룹 → 이름** 2단 피커.

| 진입점 | 동작 |
|---|---|
| **표준 생성** `startCableConnection` + `CableSpecModal` | 그룹→이름 2단. 끝점은 캔버스. |
| **OPGW 경로** `OfdRoutePopover` | 대국 OFD + 코어수 + **종류 피커**. 슬롯 2개 끝점 자동 고정, IN-IN 구조는 `buildRouteCreate`가 박음. (`CBL-OPGW` 하드코딩 제거.) |
| **선번장 코어↔설비** `EquipmentSelectCell` | **드롭다운 제거 → 읽기전용 설비명.** 연결 생성은 `SlotPortsPanel`의 "케이블 연결"(표준 흐름). (`CBL-OPJ` 하드코딩 제거.) |
| **연결탭** `CableInspector` | 종류 변경 = 그룹→이름 2단 피커(현 평면 목록 교체). |
| **계통 규격셀** `SpecCell` | 같은 그룹 안 이름 선택(현행 유지). |

**`Cable.label` 완전 제거** — 표시 이름은 항상 `category.name`. 정리: `ConnectionOverlay`(`cable.label || categoryName` → `categoryName`), `cableToLocal`, `CableInspector` 라벨행, 백엔드 `cable.service` DTO, 스키마 `Cable.label`.

## 6. 케이블 종류 관리 UI (자산관리 — 케이블종류 탭)

> **선행 의존:** "자산관리 화면"(헤더 전역 메뉴·관리자 전용·A안 드래프트+저장 셸 + 탭 구조)은 [자산관리 spec](2026-06-24-자산관리-종류체계-통일-design.md)의 P4로 설계됐으나 **아직 미구현**(현재 P1 백엔드 + P2 자산 분류만 완료). C4(케이블 종류 CRUD)는 그 셸 위에 얹히므로, **셸 + 자산종류 CRUD(자산 spec P4)를 먼저(또는 함께)** 구현해야 한다. 본 spec의 C4는 그 케이블종류 탭 내용만 정의한다.

기존 자산관리 화면(헤더 전역·관리자 전용·A안 드래프트+저장)의 **케이블종류 탭**:
- **그룹 CRUD**: 이름 + 색 추가/편집/삭제. 사용 중 삭제 FK 가드.
- **그룹 내 이름 CRUD**: 그룹 선택 → 케이블 이름 추가/편집/삭제. 사용 중 삭제 FK 가드.
- 드래프트+저장(자산종류와 동일 패턴), 관리자 전용, CONVENTIONS.md 준수(DetailCard·EditableField·Button 등 재사용).

---

## 7. 백엔드

- **스키마:** `CableGroup` 모델 신설 + `CableCategory.groupId` FK + 위 필드 제거 + `Cable` 의 `cableType`/`color`/`label` 드롭 + `CableType` enum 제거.
- **CRUD API:** `CableGroup` GET/POST/PATCH/DELETE(admin, FK 가드), `CableCategory` 쓰기(admin, groupId 필수, FK 가드) — 자산측 패턴 재사용.
- **DTO:** 케이블 응답에서 `cableType`/`code`/`displayGroup`/`displayColor` 제거, `categoryId`(+ group 색/이름 파생) 제공. `buildCableSpecification`/`specTemplate` 제거(미사용 확인됨, specification = category.name).
- **시드:** 현 `displayGroup` 5종(광/전원/네트워크/제어/접지) → `CableGroup` 5개(이름+색)로 백필, 각 카테고리 `displayGroup`→`groupId`, `Cable.cableType`→(드롭 전) 무관. fiber 시드(OPGW/OPJ)도 categoryId 유지.

---

## 8. 비목표

- `specParams` 구조 변경(코어/전압 도메인 데이터 — 불변).
- 자산측 role 체계(P2 완료) 재변경.
- 케이블 인스턴스 단위 색/라벨(제거 대상이지 재설계 아님).

---

## 9. 리스크 / 결정

- **트레이스 회귀:** cableType→group/구조 전환은 토폴로지·OPGW·코어 추적에 민감 → 기존 vitest 전수 통과가 게이트. 단계적(추가→전환→드롭).
- **IN-IN = OPGW 전용 가정:** 확인됨(전원 피더는 OUT-IN). 이 가정이 깨지는 케이블이 생기면 OPGW 판별에 슬롯-role(`role==='slot'` 양끝) 조건 추가.
- **비가역 컬럼 드롭:** cableType/color/label/code 드롭 전 백필·소비처 0 검증(2단계 마이그레이션).

---

## 10. 구현 단계 (plan에서 분할)

- **C1** 백엔드: `CableGroup` 테이블+CRUD + `CableCategory.groupId` + 백필 + DTO(group 색/이름)
- **C2** 프론트 트레이스: `cableType`→그룹/구조 전환 + 색 그룹화 (회귀 게이트)
- **C3** 생성·편집 진입점: OPGW 종류피커 · 선번장 드롭다운 제거 · CableInspector 2단 · `Cable.label` 제거
- **C4** 케이블 종류 CRUD UI(자산관리 케이블종류 탭): 그룹+이름
- **C5** 구컬럼/enum 드롭 마이그레이션 + 데드코드 0 (`cableType`·`CableType`·`code`·색맵·`getCableTypeFromMaterial` 등)
